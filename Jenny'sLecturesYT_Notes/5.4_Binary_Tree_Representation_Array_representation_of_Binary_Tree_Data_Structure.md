## 📌 Executive Summary

This lecture covers how to represent a binary tree using an **array** — index formulas for parent, left child, right child — and when array representation is efficient vs wasteful.

---

## 📐 Index Formulas (1-indexed)

For a node at index **i**:
```
Parent(i)      = i / 2         (integer division)
Left_Child(i)  = 2 * i
Right_Child(i) = 2 * i + 1
```

For **0-indexed** arrays:
```
Parent(i)      = (i - 1) / 2
Left_Child(i)  = 2 * i + 1
Right_Child(i) = 2 * i + 2
```

---

## 💻 Example (1-indexed)

```
Tree:           Array:
       A        Index: 1  2  3  4  5  6  7
      / \       Value: A  B  C  D  E  F  G
     B   C
    / \ / \
   D  E F  G
```

| Node | Index | Left Child | Right Child | Parent |
|------|-------|-----------|-------------|--------|
| A | 1 | 2(B) | 3(C) | — |
| B | 2 | 4(D) | 5(E) | 1(A) |
| C | 3 | 6(F) | 7(G) | 1(A) |

---

## ⚠️ When Array Representation Wastes Space

For **skewed/degenerate** trees:
```
Tree:    A          Array: [A, B, _, C, _, _, _, D]
          \          Index:  1  2  3  4  5  6  7  8
           B
            \       4 nodes but need 8 array positions!
             C      50% space wasted
              \
               D
```

**Worst case**: n nodes can need up to **2^n − 1** array positions.

---

## 📊 When to Use Array vs Linked Representation

| Aspect | Array | Linked (Pointers) |
|--------|-------|-------------------|
| Best for | **Complete/Perfect** trees | Any tree |
| Space | O(2^h) worst | O(n) always |
| Access | O(1) by index | O(n) traversal |
| Used in | **Heaps** | BST, AVL, etc. |

---

## 💡 Key Takeaways

- Array is **ideal for complete binary trees** (no wasted space)
- **Heaps** always use array representation
- Skewed trees waste massive space in arrays → use pointers
- Parent = i/2, Left = 2i, Right = 2i+1 (memorize!)
