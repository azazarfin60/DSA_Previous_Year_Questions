## 📌 Executive Summary
Deletion in a B+ Tree shares the same structural rebalancing mechanics as a standard B-Tree (checking for underflow, borrowing from siblings, and merging). 

However, because of the fundamental rule of B+ Trees—**all actual data must reside in the leaf nodes**—the deletion process requires extra care to maintain the internal "index" nodes. If you delete data from a leaf, you must ensure you don't leave a "dead" index pointing to it in the internal nodes.

---

## ⚙️ The Core Rules of B+ Tree Deletion

Whenever you are asked to delete a key, **always start your deletion at the leaf node**. 

### 1. Simple Deletion (No Underflow)
If the leaf node has more than the minimum required keys:
- Delete the key from the leaf node.
- **Index Check:** Look up the tree. If the deleted key also existed in an internal node as an index, replace that index with the **new minimum value** of its right subtree.

### 2. Underflow: Borrowing from Siblings
If deleting the key causes the leaf node to drop below the minimum required keys:
- **Check Right Sibling:** If the right sibling has extra keys, borrow its minimum key.
- **Check Left Sibling:** If the left sibling has extra keys, borrow its maximum key.
- **Index Update:** Because the boundaries between the nodes have changed, you must update the parent's index to reflect the new minimum value of the right child.

### 3. Underflow: Merging Leaf Nodes
If neither sibling has extra keys to spare:
- Merge the underflowing leaf node with its immediate sibling.
- **Index Deletion:** Unlike a standard B-Tree where the parent key drops down to join the merged nodes, in a B+ Tree leaf merge, the parent index separating them is simply **deleted**. (It is no longer needed because the two nodes are now one).

### 4. Cascading Merges (Internal Nodes)
If deleting an index from the parent causes the *parent* (an internal node) to underflow:
- The internal node must borrow or merge.
- **Rule Difference:** When merging *internal* nodes, it behaves exactly like a standard B-Tree. The parent's key *does* drop down to join the merged internal nodes.

## 📝 Exam Trap: Index Updating

> [!WARNING]
> **The Most Common Mistake:**
> Students often successfully rebalance the leaf nodes (borrowing/merging) but forget to update the internal index pointers. 
> 
> **Always trace upwards!** If a leaf node's first element changes due to a deletion or a borrow operation, the parent's index pointing to that node *must* be updated to match the new first element.
