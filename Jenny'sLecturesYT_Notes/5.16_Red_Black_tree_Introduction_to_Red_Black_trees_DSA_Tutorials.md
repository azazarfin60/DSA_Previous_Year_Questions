## 📌 Executive Summary
This lecture introduces Red-Black Trees, explaining why they are needed compared to standard BSTs and AVL trees. It covers the 6 strict properties that define a Red-Black Tree and explains how they ensure the tree remains roughly balanced to guarantee `O(log n)` time complexity for operations.

---

## 🆚 Why Red-Black Trees? (vs BST & AVL)

| Tree Type | Issue / Advantage |
| :--- | :--- |
| **Standard BST** | Can become completely skewed (e.g., all right children), degrading time complexity to **O(n)** worst-case. |
| **AVL Tree** | Strictly height-balanced. Searching is very fast (strictly O(log n)). However, insertion and deletion may require **many rotations** all the way up to the root, which is costly. |
| **Red-Black Tree** | Roughly height-balanced. Guarantees **O(log n)** time. The major advantage is that balancing requires **at most 2 rotations** (plus simple recoloring). |

> **Rule of Thumb:**
> - Use **AVL Trees** if your application is strictly read-heavy (frequent searching).
> - Use **Red-Black Trees** if your application requires frequent insertions and deletions.

---

## 🔑 The 6 Properties of Red-Black Trees

For a tree to be a valid Red-Black Tree, it must satisfy **ALL** of the following properties:

1. **BST Property:** It must be a valid Binary Search Tree (left subtree < node < right subtree).
2. **Color Property:** Every node is colored either **Red** or **Black** (usually requiring just 1 extra bit per node).
3. **Root Property:** The Root node is **always Black**. *(Analogy: The head of the family should always remain calm).*
4. **Leaf Property:** Every leaf node (the external `NIL` nodes at the bottom of the tree) is considered **Black**.
5. **Red Node Property:** If a node is **Red**, then both of its children **must be Black**. 
   - *Consequence:* You can never have two adjacent Red nodes (no Red-Red parent-child relationship).
6. **Black Depth Property:** Every path from a node to any of its descendant `NIL` leaf nodes must contain the **exact same number of Black nodes**.

---

## 📐 Derived Property: Path Length Limit

Because of Property 5 (no two adjacent reds) and Property 6 (same number of black nodes on every path), we get a crucial derived mathematical guarantee:

> **The longest path from the root to any leaf is no more than twice the length of the shortest path.**

**Why?**
- The shortest possible path contains only Black nodes.
- The longest possible path alternates between Black and Red nodes (e.g., B-R-B-R-B).
- Since every path must have the *same* number of Black nodes, the alternating path can at most insert one Red node between every Black node, making it exactly twice as long as the all-Black path.

---

## ⚠️ Common Violations (Exam Examples)

When asked to identify if a tree is a Red-Black Tree, check these in order:
1. Is the root red? → **Not an RBT** (Violates Property 3).
2. Are there two red nodes connected? → **Not an RBT** (Violates Property 5).
3. Is it a valid BST? (e.g., a left child is larger than its parent) → **Not an RBT** (Violates Property 1).
4. Count the black nodes on all paths from the root to the `NIL` leaves. Are they different? → **Not an RBT** (Violates Property 6).

*Note: A perfect binary tree where every single node is Black is a valid Red-Black Tree!*
