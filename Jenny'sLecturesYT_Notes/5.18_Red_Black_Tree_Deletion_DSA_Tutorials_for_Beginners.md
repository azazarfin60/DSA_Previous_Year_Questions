## 📌 Executive Summary

Red-Black Tree deletion is more complex than insertion. It handles **double-black** situations that arise when deleting a black node, using recoloring and rotations.

---

## 📐 Deletion Steps

1. Perform **standard BST deletion** to find replacement
2. If deleted node or its replacement is **Red** → simple (just recolor to Black)
3. If both deleted node and replacement are **Black** → **Double Black** problem

---

## 📐 Double Black Cases

When a double black (DB) node exists, check its **sibling**:

### Case 1: Sibling is Black, at least one Red child
- Perform **rotation** (like AVL — LL, RR, LR, RL based on red child position)
- Recolor appropriately
- DB removed!

### Case 2: Sibling is Black, both children are Black
- **Recolor** sibling to Red
- Push DB **up** to parent
- If parent was Red → make it Black (done!)
- If parent was Black → parent becomes DB (repeat)

### Case 3: Sibling is Red
- **Swap colors** of parent and sibling
- **Rotate** parent toward DB
- This converts to Case 1 or Case 2
- Apply appropriate case

---

## 📐 Summary Table

| Sibling Color | Sibling's Children | Action |
|--------------|-------------------|--------|
| Black | ≥1 Red child | Rotate + Recolor |
| Black | Both Black | Recolor sibling Red, push DB up |
| Red | — | Swap colors + Rotate → reduce to above |

---

## ⚠️ Special Cases
- **Deleted node is Red** → no DB, no fix needed
- **Root becomes DB** → simply make it single Black
- **Replacement is Red** → recolor to Black

---

## 💡 Key Takeaways

- Deletion creates **Double Black** when both deleted and replacement are Black
- Three sibling cases to handle DB
- Case 3 (Red sibling) always reduces to Case 1 or 2
- Root DB → just remove extra black
- More complex than AVL deletion but fewer rotations overall
