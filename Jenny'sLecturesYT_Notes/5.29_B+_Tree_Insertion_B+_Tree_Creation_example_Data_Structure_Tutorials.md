## 📌 Executive Summary
A **B+ Tree** is an advanced variation of a B-Tree designed specifically to optimize disk storage and database indexing. 

There are two major architectural differences between a standard B-Tree and a B+ Tree:
1. **Data vs. Indexes:** In a B-Tree, actual data is stored in all nodes. In a B+ Tree, **actual data is stored ONLY in the leaf nodes**. Internal nodes contain only "indexes" (pointers) to guide the search.
2. **Linked Leaves:** All leaf nodes in a B+ Tree are connected via a Linked List pointer, allowing for extremely fast sequential range queries.

---

## ⚙️ The B+ Tree Splitting Algorithm (The Duplication Rule)

The parameters for Order $M$ (Max children, Max keys) remain exactly the same as a B-Tree. However, the splitting behavior is different depending on whether you are splitting a **Leaf Node** or an **Internal Node**.

### 1. Splitting a Leaf Node (Data Duplication)
Because *all* data must reside in the leaf nodes, when a leaf node overflows and its median element is pushed up to the parent, **a copy of that median element must remain in the leaf level**.
- *Example:* Leaf node `[1 | 4 | 7 | 10]` overflows. Median `7` is pushed up to the parent.
- *Result:* Parent gets `7`. Left leaf becomes `[1 | 4]`. Right leaf becomes `[7 | 10]`. (Notice how 7 exists in both the parent and the right leaf).

### 2. Splitting an Internal Node (No Duplication)
Internal nodes hold indexes, not data. Therefore, when an internal node overflows, it behaves exactly like a standard B-Tree: the median element is pushed up and **is NOT duplicated**.

## 📝 Exam Walkthrough: B+ Tree Order 4 Insertion

> **Task:** Construct a B+ Tree of Order 4 inserting the keys `1, 4, 7, 10, 17, 21, 31...`. Assume a **Right-Biased** tree (as M=4 is even, we pick the right-middle element for the median).

**Parameters:** Order $M = 4$. Max keys = 3. 

**1. Insert 1, 4, 7:**
```text
[ 1 | 4 | 7 ]
```

**2. Insert 10 (Leaf Overflow):**
Node becomes `[1 | 4 | 7 | 10]`. Right-Biased median is **7**.
- Push **7** up.
- Left leaf: `[1 | 4]`. Right leaf: `[7 | 10]`.
```text
      [ 7 ]
     /     \
  [1|4] -> [7|10]
```
*(Note the arrow `->` representing the linked list connection between leaf nodes).*

**3. Insert 17, 21 (Leaf Overflow):**
Insert into the right leaf: `[7 | 10 | 17 | 21]`. Median is **17**.
- Push **17** up. Right leaf retains 17.
```text
      [ 7 | 17 ]
     /    |     \
  [1|4]->[7|10]->[17|21]
```

**4. Insert 31, 25 (Leaf Overflow):**
Insert 31, then 25 into the right leaf: `[17 | 21 | 25 | 31]`. Median is **25**.
- Push **25** up. Right leaf retains 25.
```text
       [ 7 | 17 | 25 ]
      /    |    |     \
 [1|4]->[7|10]->[17|21]->[25|31]
```

**5. Internal Node Overflow:**
Assume we continue inserting, and a new median (say, `20`) is pushed up from the leaves to the parent root `[ 7 | 17 | 25 ]`.
- Root becomes `[ 7 | 17 | 20 | 25 ]`. This is an **Internal Node Overflow**.
- Right-Biased Median is **20**.
- Push **20** up to a new root.
- **CRITICAL:** Because this is an *internal* node, `20` is NOT retained in the child.
- *Result:* New root `[20]`. Left child index node `[7 | 17]`. Right child index node `[25]`.
