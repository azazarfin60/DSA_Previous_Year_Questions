# 5.28 B-Tree Deletion | Data Structures

📺 [Watch on YouTube](https://www.youtube.com/watch?v=115K8LwP-xY) *(Note: Placeholder link)*

---

## 📌 Executive Summary
Deletion in a B-Tree is significantly more complex than insertion because it can cause a node to **underflow** (drop below the minimum required keys $\lceil M/2 \rceil - 1$). 

The deletion algorithm must constantly check for underflow. If underflow occurs, the tree must repair itself by either **Borrowing** from immediate siblings or **Merging** nodes together, which can cascade upwards and eventually shrink the height of the tree.

---

## 🗑️ Case 1: Deleting from a Leaf Node

When the target key is located in a leaf node, we check the number of keys currently in that leaf node.

### Subcase 1A: Node has > Minimum Keys
This is the simplest scenario. 
- **Action:** Simply delete the key. The B-Tree properties are maintained.

### Subcase 1B: Node has exactly Minimum Keys (Underflow)
Deleting the key will violate the B-Tree properties. The node must seek help.

1. **Borrow from Left Sibling:** 
   - *Condition:* The immediate left sibling has > minimum keys.
   - *Action:* Borrow the **Maximum** key from the left sibling. It moves up to the parent, and the parent's separating key moves down into the target node.
2. **Borrow from Right Sibling:** 
   - *Condition:* Left sibling cannot help, but the immediate right sibling has > minimum keys.
   - *Action:* Borrow the **Minimum** key from the right sibling. It moves up to the parent, and the parent's separating key moves down into the target node.
3. **Merge (If neither sibling can help):**
   - *Action:* Merge the target node with one of its immediate siblings **AND** the parent key that separates them. 
   - *Warning:* Because a key is pulled down from the parent, the parent might now underflow! If so, recursively apply these rules to the parent.

---

## 🗑️ Case 2: Deleting from an Internal Node

You cannot simply delete an internal node key, because it acts as a separator for its children. You must replace it with a key from its subtrees.

### 1. Replace with In-order Predecessor
- **Definition:** The maximum element in the target key's left subtree.
- **Condition:** The leaf node containing the predecessor must have > minimum keys.
- **Action:** Replace the target key with the predecessor, and delete the predecessor from its leaf node.

### 2. Replace with In-order Successor
- **Definition:** The minimum element in the target key's right subtree.
- **Condition:** Predecessor failed, but the leaf node containing the successor has > minimum keys.
- **Action:** Replace the target key with the successor, and delete the successor from its leaf node.

### 3. Merge Children
- **Condition:** Neither the predecessor's nor the successor's node can spare a key.
- **Action:** Merge the left child and right child of the target key, and push the target key down into this new merged node. Then, delete the target key from the newly merged leaf node.

---

## 📉 Shrinking the Tree
During a complex merge operation, the root node might have its only key pulled down to facilitate a merge between its two children. 

When the root becomes entirely empty, it is deleted. Its newly merged single child becomes the new root of the B-Tree, effectively **decreasing the height of the tree by 1**.
